## Overview

Parkomat.Agent is a **Windows-only .NET 10 / C#** agent that runs on a PC at a parking
site. It reads a PLC over **Modbus-TCP** and relays the site's state to the cloud
(**HiveMQ**) over **MQTT**, through a local **Mosquitto** bridge.

End-to-end data flow:

```
PLC â†’ (Modbus-TCP) â†’ Agent Service â†’ (MQTT localhost:1883) â†’ Mosquitto â†’ (bridge/TLS) â†’ HiveMQ â†’ server
```

## Build / run / publish

Run from the `Parkomat.Agent/` directory:

```sh
# Build the whole solution
dotnet build Parkomat.Agent.slnx

# Run the Service locally as a console app (dev; DOTNET_ENVIRONMENT=Development via launchSettings)
dotnet run --project src/Parkomat.Agent.Service

# Run the Tray app
dotnet run --project src/Parkomat.Agent.Tray

# Run tests
dotnet test Parkomat.Agent.slnx

# Publish for deployment â€” output paths MUST match what installer.iss expects
# (publish/service, publish/tray). MUST be SELF-CONTAINED: site PCs are not
# guaranteed to have the .NET 10 Desktop runtime installed, and a framework-
# dependent build simply won't start there â€” the Tray never appears and it looks
# identical to "install failed". A self-contained build bundles the runtime
# (installer ~64MB; a framework-dependent one is ~5MB â€” that size drop is the
# red flag). Always publish with -r win-x64 --self-contained before compiling
# the installer.
dotnet publish src/Parkomat.Agent.Service -c Release -r win-x64 --self-contained true -o publish/service
dotnet publish src/Parkomat.Agent.Tray    -c Release -r win-x64 --self-contained true -o publish/tray
```

Installer: **Inno Setup** compiles `installer.iss` â†’ `installer-output/ParkomatAgentSetup.exe`.

Tests live in `tests/Parkomat.Agent.Service.Tests` (xUnit). Current coverage is the
`OperationDetector` state/operation logic. For behavior with no unit coverage (PLC I/O, MQTT),
verify by running the Service/Tray directly.

## Runtime data locations

All paths come from `AgentPaths` (`src/Parkomat.Agent.Core/Configuration/AgentPaths.cs`).
Base folder is `C:\ProgramData\Parkomat\Agent\` â€” shared so both the Service (runs as SYSTEM)
and the Tray (runs as the user) can read/write it:

- `config.json` â€” site + PLC + MQTT settings (`ConfigStore` / `SiteConfig`)
- `heartbeat` â€” unix-seconds timestamp written after each successful PLC read
- `bridge.conf` â€” Mosquitto bridge config, generated by `BridgeConfigWriter`
- `agent.db` â€” reserved (declared path; not yet used)

## Architecture & control flow

`Worker.cs` (`ExecuteAsync`) runs the main loop: **read PLC â†’ feed the brain â†’ publish**.

- **`OperationDetector`** is the pure "brain": it *decides* what to publish and never sends
  anything itself. It remembers the previous MODE in `_previousMode` and emits a `state`
  message when the translated state changes, plus `operation` start/end messages on
  entry/exit transitions.
- **PLC failure handling:** 10 consecutive failed reads â†’ publish `state: error` **once**
  (guarded by `plcErrorReported`), reset when the PLC recovers.
- **Heartbeat:** written on every successful read. The Tray (`ServiceManager`) reads it back
  to distinguish "really working" (heartbeat fresh â‰¤ 10s â†’ colored icon) from merely
  "process alive" (service running but no fresh heartbeat).

## MQTT / server contract (non-obvious â€” easy to break)

These are enforced by the server; changing them silently breaks ingestion:

- **Topics:** `sites/{siteId}/state`, `sites/{siteId}/operation`, and `sites/{siteId}/bridge`
  (the last one is published by **Mosquitto**, not by us â€” see "Disconnect detection" below).
- **Timestamps are unix _seconds_, not milliseconds.** The server floors its own timestamps to
  whole seconds to match; a mismatch here strands sites in `no_comm` permanently.
- **`SiteState` JSON strings are exact and server-enforced:** `ready`, `operating`, `error`,
  `maintenance`, `no_comm` (see `SiteState.cs` `[JsonStringEnumMemberName]`). Do not rename.
- **Operation messages must include a valid `state`** or the server rejects them.
- **`user` (card number) must be `""`, never `null`** â€” it is part of the server's dedup key.
- **The Service always connects to local Mosquitto (`localhost:1883`), never directly to
  HiveMQ.** The `MqttConfig` host/port/username/password feed the generated `bridge.conf`
  (`BridgeConfigWriter`) â€” they are *not* the client connection settings.

## Disconnect detection â€” two LWT layers (do not collapse them)

There is no heartbeat and no server-side watchdog. "90 seconds" is **1.5 Ă— the 60s keepalive**,
enforced by the brokers. Two layers, because one is not enough:

1. **Agent â†’ local Mosquitto.** Our LWT: JSON `{"timestamp":0,"state":"no_comm"}` on the state
   topic. Covers: *the agent process crashed while the PC kept running.*
2. **Mosquitto bridge â†’ HiveMQ.** Mosquitto's own bridge notification, payload `"1"`/`"0"` on
   `sites/{siteId}/bridge`. Covers: **the PC lost power.** In that case Mosquitto dies together
   with the agent, so *nobody local is left to publish layer 1* â€” only HiveMQ, which holds the
   bridge's will, can report it.

Layer 2 is the failure that actually matters at a car park, and it was silently broken:
`bridge.conf` had `notifications_local_only true`, so the notice stayed on a **powered-off PC**
and the server showed the site as healthy indefinitely. `BridgeConfigWriter` now sets
`notifications_local_only false` and an explicit `keepalive_interval 60`. Do not "tidy" these away.

The payload is `"1"`/`"0"`, **not JSON** â€” which is exactly why it needs its own topic and cannot
share `sites/{siteId}/state`, where the server contract demands JSON.

On bridge reconnect the agent **republishes its current state with a fresh timestamp**
(`Worker.cs`). Without that resync, messages queued during the outage carry old timestamps, the
server's backfill guard rejects them, and the site stays `no_comm` after it has already recovered.

## TLS is mandatory and cannot be switched off

`MqttConfig` has **no `UseTls` field**, and the settings form has no checkbox â€” both were
removed on purpose. The bridge carries the site's username and password to HiveMQ across the open
internet; without TLS they travel in clear text. A checkbox there was not flexibility but a trap:
one click by a technician in the field silently downgraded the whole site, and nothing failed
loudly. `BridgeConfigWriter` always emits `bridge_tls_version tlsv1.2` + `bridge_cafile`.

The CA certificate lives under `ProgramData` (no spaces in the path), not `Program Files`.
- **MODE â†’ state mapping lives only in `ModeTranslator`:** `0`=maintenance, `1`=ready,
  `2`/`3`=operating, `4`=init (ignored, no state), `5`=error. For operations, `2`=entry,
  `3`=exit.

## Conventions

- Source comments are in **Hebrew** â€” match that when editing existing files.
- `Nullable` and `ImplicitUsings` are enabled across projects.
- **Atomic file writes** (write to `.tmp`, then `File.Move(..., overwrite: true)`) in
  `ConfigStore` and `BridgeConfigWriter` â€” preserve this pattern for any config the service
  writes, so a power loss mid-write can't corrupt the file.

## Deployment notes

`installer.iss` is a **per-user, no-admin** install (`PrivilegesRequired=lowest`, no UAC
prompt). It does **not** register any Windows service and does **not** use `nssm`. Everything
runs in the user's session:

- The install goes to `{localappdata}\Parkomat\Agent` (service + tray + bundled Mosquitto).
- The **Tray** auto-starts per-user via an **`HKCU\...\Run`** registry value (not `HKLM`).
- The Tray is the supervisor: it launches and watchdogs **both** the Agent process and
  **Mosquitto** as ordinary child processes (no services). Runtime data still lives under
  the shared `C:\ProgramData\Parkomat\Agent` path (a standard user may create/own it).

> Nothing in install or runtime requires administrator rights. `PrepareToInstall` does make
> best-effort `sc.exe stop/delete` calls to clean up an **older admin-era** install; without
> admin those simply fail silently (no UAC, no error) and the install proceeds. Do not
> reintroduce a service-based (`nssm` / `sc create`) install â€” it would bring back the admin
> requirement this design deliberately removed.
